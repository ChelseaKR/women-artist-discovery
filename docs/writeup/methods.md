# Values-aware recommendation without inferring identity

> The engineering-ethics piece [`docs/ROADMAP.md` §9](../ROADMAP.md) calls "the
> artifact": *"The writeup is the artifact: how to do values-aware recommendation
> without inferring identity."* This document is that writeup — and, per
> [EXP-10's excellence bar](../ideation/03-expansions.md), every number in it is
> generated by a command you can run yourself, not typed in by hand.
>

> `precision@5 = 0.60 [docs/audits/eval-report.json → models.hybrid.precision_at_k]`,

## 1. Threat framing — the harm of guessing

Most "diversity-aware" recommenders that segment by gender do it the cheap way:
infer it from a first name, a voice, a promo photo, or a genre association. That
approach is attractive because it needs no cooperation from the artist and scales
to any catalog — and it is exactly the failure mode this project refuses to build.

Guessing identity from proxies causes concrete harms, independent of how accurate
the guess is on average:

- **Misgendering as a side effect of a ranking feature.** A name- or voice-based
  classifier will misgender nonbinary and trans artists at a materially higher
  rate than it misgenders cisgender artists matching stereotypical cues — the
  people a values lens is supposed to serve worst-served by the mechanism meant
  to serve them.
- **A durable, unconsented dataset.** Once you've built a classifier, you've built
  a "perceived gender of every artist in the catalog" dataset — sourced from
  nothing the artist agreed to disclose, and reusable for purposes far outside
  music discovery. See ["Non-redistribution"](../audits/identity-data-ethics.md)
  for why this repo structurally cannot produce that artifact even by accident.
- **Silent, unauditable error.** An inference has no citation. When it's wrong,
  there is nothing to point to, no fetch date, no source to correct — the error
  just sits in the ranking forever. A *sourced* label is wrong sometimes too
  (§5, RR-2), but it is wrong in a way you can find and fix.
- **Essentializing "female-fronted."** Collapsing a band's identity into one
  member's inferred gender (or worse, inferring it from timbre) turns a
  composition fact into a claim about a person. This project keeps
  `female_fronted` as sourced, tri-state *band* metadata, never a personal label
  — see [`identity-data-ethics.md`](../audits/identity-data-ethics.md).

The response is not to avoid identity-aware recommendation — a values lens is the
whole point of this project — but to make the *no-guessing* rule a property of the
type system, not a policy reviewers hope engineers remember. §2 is that rule made
mechanical.

## 2. Type-invariant approach

The guardrail lives in
[`pipeline/models.py`](../../pipeline/models.py), enforced at construction time,
not just at review time:

- **`Gender`** (`pipeline/models.py::Gender`) is a five-member controlled
  vocabulary — `WOMAN`, `MAN`, `NONBINARY`, `OTHER`, `UNKNOWN` — and `UNKNOWN` is
  not a fallback bolted on later; it is a first-class enum member and the
  dataclass default (`IdentityLabel.gender: Gender = Gender.UNKNOWN`).
- **Sourced-only identity.** `IdentityLabel.__post_init__` raises
  `UnsourcedIdentityError` if a non-`UNKNOWN` gender is constructed without at
  least one `Source`, and `InferenceForbiddenError` if that source isn't an
  *individual-identity* `SourceKind` (`WIKIDATA_P21`, `MUSICBRAINZ_GENDER`,
  `ARTIST_STATEMENT`) with a `SELF_IDENTIFIED` basis. There is deliberately no
  `SourceKind` member for a name, voice, image, or genre — the guardrail isn't
  just "don't call the inference function," it's "there is no inference function
  to call." `tests/test_no_inference.py` proves this with a vocabulary check and
  an AST scan of the resolver, not just a behavioural test.
- **`unknown` as first-class, not a null.** `IdentityLabel()` with no arguments
  *is* a valid, complete identity — `UNKNOWN_IDENTITY`, the module-level
  singleton. Nothing downstream branches on "missing data"; it branches on a
  real enum value that every exhaustive match must handle.
- **Composition kept separate from personal identity.** `BandComposition
  .female_fronted` is a sourced, tri-state (`True` / `None`) property of a
  *band*, built from `FrontPerson` entries whose *own* `identity` is itself a
  full `IdentityLabel` — never a shortcut that infers a member's gender from the
  fact that the band is described as female-fronted, or vice versa.

The segmentation vocabulary that the rest of this piece reports against —
`WOMAN` / `NONBINARY` / `FEMALE_FRONTED` / `MAN` / `OTHER` / `UNKNOWN` — is
defined once, in [`recommender/exposure.py`](../../recommender/exposure.py)
(`SEGMENTS`), and derived purely from `Artist.identity` and `Artist
.female_fronted` — i.e., purely from the sourced fields above. Segmenting never
adds a new way to guess; it only reads what §2's guardrails already sourced.

## 3. Boost-only proof

The values lens re-ranks, it never penalizes. That's implemented in
[`recommender/rerank.py`](../../recommender/rerank.py):
`values_boost_for_artist` returns `0.0` for any artist that isn't
*sourced*-aligned (including every `UNKNOWN` artist) and a non-negative boost
otherwise; `rerank()` asserts `delta >= 0.0` on every recommendation before
re-sorting. By construction, an `UNKNOWN`-identity artist's score is invariant to
`lens_strength` — it can move in *list position* only if a sourced-aligned artist
is boosted above it, never because its own score dropped.

"True by construction" is a claim about the code. It is only a credible claim
about the *product* once it's checked against what the pipeline actually emits —
which is what [`recommender/exposure.py`](../../recommender/exposure.py) is for.
`assert_unknown_retained(recs_by_lens, base_lens=0.0)` takes the *emitted*
recommendation lists at each lens strength and raises `FairnessAssertionError` if
any `UNKNOWN`-segment artist present at `lens_strength = 0` (pure taste) is either
dropped from later output or has its score lowered. `unknown_retention()` reports
the same check as a number:

> **unknown-retention = 1.0** at every lens strength `[recommender/exposure.py →
> unknown_retention(); exercised on real reranked output by
> tests/test_exposure.py::test_unknown_retention_is_1_across_lens_strengths_on_real_reranked_output]`
> — 1.0 by construction (§3, `rerank.py`), and checked on emitted output, not
> asserted in prose, every time `make audit` runs the test suite. The guard
> itself is tested too: `test_assert_unknown_retained_raises_on_a_downranked_unknown`
> and `..._raises_on_a_dropped_unknown` build synthetic violations and confirm
> `FairnessAssertionError` actually fires — the check isn't a rubber stamp.

This is the same guarantee `docs/audits/fairness-identity.md` names as "metric
*down-ranked-for-unknown = 0*" (finding 2) — `exposure.py` is what turns that
narrative claim into a value `make audit` regenerates.

## 4. Exposure numbers

The offline eval (`recommender/eval.py`, run via `make eval` →
[`pipeline/cli.py`](../../pipeline/cli.py) `eval` subcommand) temporally splits
one user's scrobbles, holds out the later plays as ground truth, and scores two
rankers — the hybrid (hybrid + lens at `lens_strength = 0`, i.e. pure taste) and a
popularity baseline — at `k = 5`. Every number below is read verbatim from the
regenerated `docs/audits/eval-report.json`:

| Model | precision@5 | recall@5 | MAP@5 |
|---|---|---|---|
| hybrid | 0.60 `[docs/audits/eval-report.json → models.hybrid.precision_at_k]` | 0.75 `[docs/audits/eval-report.json → models.hybrid.recall_at_k]` | 0.6875 `[docs/audits/eval-report.json → models.hybrid.map_at_k]` |
| popularity | 0.20 `[docs/audits/eval-report.json → models.popularity.precision_at_k]` | 0.25 `[docs/audits/eval-report.json → models.popularity.recall_at_k]` | 0.0833 `[docs/audits/eval-report.json → models.popularity.map_at_k]` |

`hybrid_beats_popularity = true` `[docs/audits/eval-report.json →
hybrid_beats_popularity]` — the merge-blocking condition `make eval` enforces
(`n_positives = 4` `[docs/audits/eval-report.json → n_positives]` held-out
discoveries in this fixture).

That primary table is the deliberately hand-tuned demo world, so the audit also
runs four independent synthetic fixture families. Across all five worlds, the
hybrid wins 5 `[docs/audits/eval-report.json → multiworld.worlds_hybrid_wins]`
of 5 `[docs/audits/eval-report.json → multiworld.n_worlds]`, with mean
MAP@5 delta 0.5375 `[docs/audits/eval-report.json →
multiworld.mean_map_delta]`. These remain synthetic tests, but they keep the
claim from resting on the one fixture designed alongside the recommender.

Read narrowly: this shows the *hybrid engine* recovers genuine future discoveries
better than "just recommend what's already popular" — it is not, on its own, a
fairness metric. The fairness claim is §3's unknown-retention number and
`fairness-identity.md`'s segment findings; §4 establishes that the underlying
taste model is worth re-ranking with a values lens in the first place, rather
than a values lens papering over a recommender nobody would use.

## 5. Honest limits

- **Source sparsity (RR-2).** Identity sources — Wikidata P21, MusicBrainz's
  gender field, cited artist statements — do not cover the catalog uniformly, and
  what they do cover can be **wrong or stale**
  `[docs/audits/residual-risk.md → RR-2]`. The project's answer is not to fill
  the gap with an inference, but to (a) let `UNKNOWN` be the honest, first-class,
  never-penalized answer (§2), and (b) hedge confidence on source conflict, show
  basis + citation + fetch date on every known label, and fold corrections back
  through re-enrichment rather than hand-editing a label in place
  (`docs/audits/residual-risk.md`, `docs/audits/identity-data-ethics.md`
  "Correctability").
- **The `UNKNOWN` segment is not small, and that's by design, not a defect.**
  Because identity is never guessed, any artist without a citable source lands in
  `UNKNOWN` — for a niche or catalog-tail artist, that can be the majority
  segment. `UNKNOWN` artists are surfaced on musical merit alone (§3); the
  tradeoff is that the values lens itself can only ever act on the sourced
  minority. Growing lens *coverage* means growing *sourced* identity coverage —
  more citable evidence — never loosening what counts as a source.
- **Allocational risk within the sourced set.** `docs/audits/fairness-identity.md`
  (finding 3) names a further risk even where sourcing is good: because the boost
  is a fixed re-rank on top of a taste-only base score, it can over-favor
  already-popular sourced-aligned artists over niche ones. The mitigation is a
  bounded boost (`MAX_BOOST`, `recommender/rerank.py`) that re-orders without
  overriding taste, and the popularity-tier × identity cross-tab that the
  fairness audit tracks — this piece does not claim that risk is resolved, only
  bounded and watched.
- **Synthetic evaluation is not population evidence.** §4 reports the demo
  temporal split and four independent synthetic fixture families. That is
  stronger regression evidence than the original single tuned world, but it is
  still not a benchmark across real listeners. The separate `make eval-real`
  leg is intentionally local and human-gated because listening histories are
  sensitive; no real user data is committed or run in CI.
