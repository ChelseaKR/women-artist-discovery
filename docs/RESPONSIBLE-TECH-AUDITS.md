# Responsible-Tech Audits — Women-Artist Discovery

> **Last verified: 2026-07-05 · Recheck cadence: per RTF/AIEV standard revision or identity-model change.**

Instantiates `/STANDARDS/RESPONSIBLE-TECH-FRAMEWORK.md`. The distinctive edge here is identity data: doing values-aware recommendation without inferring, essentializing, or building a misusable gender database. Checklists run in CI from M0; artifacts are committed and regenerated by `make verify`.

---

## A. Ethics & responsibility
- **Worst misuse:** a scraped "gender of musicians" dataset, or a tool that misgenders artists by guessing. → Identity is sourced-only, cited, correctable, and never redistributed; unknown is first-class.
- **"Works as intended" harm:** the re-rank quietly buries artists whose identity is unknown. → Unknown is never penalized (tested).
- **Non-goals:** not a guessing engine; not a gender-database product. **Auto-gated:** no-inference test; unknown-not-penalized test. **Review-gated:** identity-ethics review.

## B. Bias & fairness
- **Segments:** artists by identity basis (self-identified / band-composition / unknown) and by popularity tier.
- **Risks:** representational bias (nonbinary collapsed into a binary; unknown erased), allocational bias (re-rank over-favoring already-popular women).
- **Tests:** nonbinary representable end-to-end; unknown-retention; a popularity-debiasing check.
- **Commitment:** a genuinely non-binary identity model; surface, don't bury, unknown. **Auto-gated:** identity-model + unknown-retention tests. **Review-gated:** fairness review.

## C. Privacy & data-protection
- **Data inventory:** your Last.fm username + scrobbles (personal); cached artist metadata (public).
- **Handling:** listening history never leaves your machine; no third-party analytics or telemetry.
- **Commitment:** local-first, minimal retention, your data stays yours. **Auto-gated:** no-telemetry test; cache-scope test. **Review-gated:** privacy review.

## D. Transparency & explainability
- **Per recommendation:** the signals that produced it, the identity basis, and the *source* of that basis; data-source limitations documented in-app.
- **Commitment:** no unexplained recommendations; honest about source sparsity and uncertainty. **Auto-gated:** explanation-present + source-shown tests. **Review-gated:** framing review.

## E. Accessibility (WCAG 2.2 AA)
- **Dashboard:** keyboard-complete; charts have data-table equivalents; identity/why information is never color-only; cards are screen-reader-friendly.
- **Commitment:** zero `axe` violations plus a committed manual walkthrough. **Auto-gated:** axe + keyboard + chart-table tests. **Review-gated:** SR walkthrough sign-off.

## F. Security
- **Threat model:** API-key leakage, cache poisoning via untrusted external data, dependency risk.
- **Controls:** secrets handling, input validation on external responses, SAST + deps + secret scans merge-blocking.
- **Residual-risk register** committed in `docs/audits/`. **Auto-gated:** scanners + validation tests. **Review-gated:** threat-model sign-off.
- **ASVS level:** **ASVS 5.0 Level 1.** No auth surface, no multi-user/tenant surface, no server —
  Level 2's session/authz-heavy controls are N/A by scope; SEC §1 separately cites this repo's
  AST-level no-inference test (`tests/test_no_inference.py`) as a project-specific ASVS-V8
  (data-protection) abuse-case exemplar.
- **Container scan:** N/A — no `Dockerfile`/container image is built or shipped.
- **SAST posture:** ruff's bandit (`S`) subset is the real, merge-blocking SAST layer
  (`pyproject.toml` `[tool.ruff.lint] select`), plus CodeQL (`python` + `actions` queries,
  `.github/workflows/codeql.yml`, weekly + PR/push to main). No Semgrep — this pairing is the repo's declared
  SAST posture rather than the standard's default Semgrep expectation; revisit if a Semgrep policy
  pack becomes cheap to run at this repo's scale.
- **SBOM + signing:** prepared, not yet exercised — `.github/workflows/release.yml` generates a
  CycloneDX SBOM (`cyclonedx-py`) and cosign keyless signatures for every release artifact, but it
  only runs on a `v*` tag push and no tag exists yet (see `SECURITY.md`/`CHANGELOG.md`). Today's
  `docs/audits/vex.json` is the pre-release vulnerability-exceptions artifact, not an SBOM.
- **Secret management:** environment variables only; no secrets are committed. Enforced by
  `scripts/secret-scan.sh` locally/in `make security`, plus `gitleaks-action` as a second,
  annotating layer in CI.
- **VEX:** `docs/audits/vex.json` — CycloneDX VEX, currently an **empty, justified** waiver list
  (see `docs/audits/residual-risk.md` RR-1/RR-4 resolution history).

## AI-evaluation status (AIEV-01)
In scope per `AI-EVALUATION-STANDARD.md` §0 as a classical-ML recommender: a hybrid
(collaborative + content) model with a values-aware boost-only re-rank, no LLM SDK, no RAG, no
generation, no judge anywhere (verified — the only third-party runtime deps are `requests`,
`numpy`, `streamlit`, `pandas`; `grep` for any model-API client returns nothing). Consequently:

- §1–3 (LLM/RAG/prompt-injection/judge gates) are **dormant** — nothing there applies until this
  repo imports an LLM SDK.
- The gates that **are** active and merge-blocking today: the offline eval must beat the popularity
  baseline *and* must not regress vs `docs/audits/eval-baseline.json` (`make eval` / `wad eval`; CI
  `make verify` step; `docs/audits/eval-report.json`).
- The first LLM SDK import anywhere in this repo flips this status to full `APPLIES` and activates
  §1–3 for real.
- Full model documentation: `docs/audits/model-card.md` (AIEV-22) and `docs/audits/data-card.md`
  (AIEV-23). Consolidated risk register + AI-system impact assessment + EU AI Act classification:
  `docs/audits/ai-risk-register.md` (RTF-09/RTF-10/RTF-12).
